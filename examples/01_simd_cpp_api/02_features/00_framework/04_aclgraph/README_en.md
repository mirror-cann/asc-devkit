# ACLGraph Example

## Overview

When a fixed data copy and computation process must be executed repeatedly, the host side submits runtime tasks such as data copies, kernel launches, and synchronization before each execution by default. As the number of repetitions increases, this task dispatch overhead affects overall execution efficiency.

ACLGraph can capture a group of runtime tasks on a specified stream, and then execute these tasks repeatedly as a whole. The process is as follows: call `aclmdlRICaptureBegin` and `aclmdlRICaptureEnd` to generate an `aclmdlRI` execution instance. Then call `aclmdlRIExecuteAsync` to execute the instance without submitting the same tasks one by one again. For more information about ACLGraph, refer to [ACLGraph Documentation](https://www.hiascend.com/document/detail/zh/CANNCommunityEdition/910beta1/programug/acldevg/runtime_doc_dev_0045.html).

This example uses the Add operator to show how to capture Ascend C `<<<>>>` kernel function invocations and the related data copy tasks. It demonstrates single-stream linear task capture and double-stream event dependency capture.

**Scenario 1: Single-stream capture**
- Input: x [8, 2048], float; y [8, 2048], float
- Output: z [8, 2048], float
- Captured content: x and y input data copies, the `add_custom` kernel launch, and the z output data copy are submitted to `mainStream` in sequence.
- Description: The `aclmdlRI` generated by `aclmdlRICaptureEnd` stores a linear task chain. Then `aclmdlRIExecuteAsync` replays the whole chain. This scenario demonstrates the basic "capture once, execute repeatedly" process.

- Captured content: `mainStream` handles the x input data copy and the z output data copy. `computeStream` waits for the x copy to complete, and then performs the y input data copy and launches the `add_custom` kernel. - Description: `aclmdlRI` records task order on a single stream and event dependencies across streams. This scenario demonstrates how to capture multi-stream task relationships.

## Example Implementation

### Operator Implementation

The core content of the operator implementation is in `add.asc`. The Add computation formula is:

$$
z = x + y
$$

- x: input, shape [8, 2048], data type float;
- y: input, shape [8, 2048], data type float;
- z: output, shape [8, 2048], data type float;

Each block in the kernel function processes 2048 float elements, and 8 blocks process all input data. Each block first copies x and y from GM to UB, calls `AscendC::Add` to perform vector addition, and then writes the result back to GM.

### ACLGraph Capture Process

In this example, ACLGraph means "capture once and execute repeatedly." The host side first calls `aclmdlRICaptureBegin` to enter the capture state, submits the runtime tasks to be recorded, and then calls `aclmdlRICaptureEnd` to end capture and obtain `aclmdlRI`.

The capture process is as follows:

1. Start capture: Call `aclmdlRICaptureBegin` to start capturing runtime tasks on the specified stream.
2. Submit tasks: Submit input data copies, the kernel launch, the output data copy, and event synchronization tasks in the double-stream scenario.
3. End capture: Call `aclmdlRICaptureEnd` to end capture and generate `aclmdlRI`.
4. Execute repeatedly: Call `aclmdlRIExecuteAsync` to execute the captured task process repeatedly.
5. Release resources: Call `aclmdlRIDestroy` to release the `aclmdlRI` execution instance when it is no longer needed.

```cpp
aclmdlRICaptureBegin(mainStream, ACL_MODEL_RI_CAPTURE_MODE_GLOBAL);

// Submit input data copies, kernel launches, output data copies, event synchronization,
// and other runtime tasks.

aclmdlRICaptureEnd(mainStream, &modelRI);
```

After `modelRI` is generated, call `aclmdlRIExecuteAsync` to execute the captured task process repeatedly.

```cpp
for (int i = 0; i < 5; ++i) {
    aclmdlRIExecuteAsync(modelRI, mainStream);
    aclrtSynchronizeStream(mainStream);
}
```

The example also calls `aclmdlRIDebugJsonPrint` to output the captured task information to JSON files. The single-stream scenario outputs `modelRI_single_stream.json`, and the double-stream scenario outputs `modelRI_double_stream.json`.

## Build and Run

- Configure environment variables

  Configure environment variables based on the [installation method](../../../../../docs/quick_start.md#prepare&install) of the CANN development kit in the current environment.

  ```bash
  source ${install_path}/cann/set_env.sh
  ```

  > **Note:** `${install_path}` is the CANN package installation directory. When no installation directory is specified, the default installation path is `/usr/local/Ascend`.

- Run the example

  Run the following steps in the root directory of this example.

  ```bash
  mkdir -p build && cd build;
  cmake -DCMAKE_ASC_ARCHITECTURES=dav-2201 ..;make -j;
  ./demo 1
  ```

  `./demo 1` runs the single-stream capture scenario. `./demo 2` runs the double-stream capture scenario. If no runtime parameter is specified, the single-stream capture scenario is run by default.

- Build option description

  | Option | Values | Description |
  |--------|--------|-------------|
  | `CMAKE_ASC_ARCHITECTURES` | `dav-2201` (default), `dav-3510` | NPU architecture: dav-2201 corresponds to Atlas A2 Training Series Products/Atlas A2 Inference Series Products and Atlas A3 Training Series Products/Atlas A3 Inference Series Products; dav-3510 corresponds to Ascend 950PR/Ascend 950DT |

- Runtime parameter description

  | Parameter | Values | Description |
  |-----------|--------|-------------|
  | `mode` | 1, 2 | Capture scenario number: 1 indicates single-stream capture, and 2 indicates double-stream capture. If this parameter is not specified, the default value is 1 |

- Execution result

  The following output is observed upon success:

  ```bash
  Output: 0 0.3 0.6 0.9 1.2 ...
  Golden: 0 0.3 0.6 0.9 1.2 ...
  test pass!
  ```
