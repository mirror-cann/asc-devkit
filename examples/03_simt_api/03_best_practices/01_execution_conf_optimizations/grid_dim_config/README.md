# 核函数的启动配置性能调优样例

## 概述

本样例以Gather算子为例，在不同数据量场景下，展示不同的线程块配置策略对算子性能的影响，并提供相应的优化指导。

## 支持的产品

- Ascend 950PR/Ascend 950DT

## 支持的CANN软件版本

- \>= CANN 9.1.0

## 样例描述

- 计算公式：

  $$
  output[i][j] = input[index[i][j]]
  $$

  - input为输入张量，形状为[2100000]，数据类型为float
  - index为索引张量，数据类型为uint32，索引范围为[0,2100000)；大数据量场景为[1024,2048]，小数据量场景为[8,2048]
  - output为输出张量，数据类型为float；大数据量场景为[1024,2048]，小数据量场景为[8,2048]
  - 计算过程：根据index[i][j]索引从input中采集对应位置的数据，写入output[i][j]位置

## 样例实现

### Case实现说明

本样例分为3类Case，先用空核函数检测不同线程块数量下的启动与调度开销，再分别观察大数据量场景和小数据量场景下的线程块数量选择。

| Case   | 数据量 / 规模                      | 核函数                                  | 对比目的                                  |
| ------ | --------------------------------- | --------------------------------------- | ----------------------------------------- |
| Case 0 | 大数据量场景，[1024,2048]=2097152      | empty_kernel                            | 对比线程块数量为1 / 32 / 64 / 128 / 1024时的空核调度开销 |
| Case 1 | 大数据量场景，[1024,2048]=2097152      | gather_naive_kernel / gather_strided_kernel | 对比线程块数量为1024 / 64 / 32时的性能差异 |
| Case 2 | 小数据量场景，[8,2048]=16384           | gather_strided_kernel                   | 对比线程块数量为4 / 8 / 16 / 32 / 64时的性能差异 |

### 性能指标说明

| 指标                | 说明                                                                                      |
| ------------------- | ----------------------------------------------------------------------------------------- |
| Task Duration（us）   | Task整体耗时，包含调度到加速器的时间、加速器上的执行时间以及响应结束时间                  |

### Case 0：空核函数的启动调度开销

**样例目标**：用空核函数检测不同线程块数量下的启动调度开销。

**核心实现**：

- 核函数`empty_kernel`函数体为空
- 通过`SCENARIO_NUM`取值为1~5控制`<<<...>>>`设置五种线程块数量配置：1 / 32 / 64 / 128 / 1024

**关键代码**：

```cpp
__global__ __launch_bounds__(MAX_THREAD_NUM) void empty_kernel(
    float* input, uint32_t* index, float* output,
    uint32_t input_total_length, uint32_t index_total_length)
{
}

// 线程块数量 由 SCENARIO_NUM 选择：1 / 32 / 64 / 128 / 1024，每线程块线程数固定2048
empty_kernel<<<blocks_per_grid, MAX_THREAD_NUM, 0, stream>>>(
    input_device, index_device, output_device,
    input_total_length, index_total_length);
```

**性能数据**：

<img src="figures/fig0_case0_empty_curve.png" alt="Case 0 空核调度基线 Task Duration 随线程块数量的变化" width="600" />


**分析**：

从上图可以看出，在空核场景下，随着线程块数量从1增长到1024，Task Duration也随之增长：线程块数量为1时Task Duration为1.221us，增加到32时为1.557us，增加到64时为2.173us，增加到128时为3.141us，增加到1024时进一步增长到16.422us。

**结论**：

由于空核本身几乎不做计算，这部分耗时主要体现为启动和调度线程块带来的固定开销。由此可见，线程块数量越多，启动和调度线程块固定开销越大，整体Task Duration也会相应增加。

### Case 1：大数据量场景，Gather算子配置不同线程块数量性能对比

**样例目标**：对比同一大数据量场景`1024*2048`，即2097152个元素下，执行Gather算子，配置不同线程块数量的性能差异。

#### 单线程单数据

**核心实现**：

- Gather算子计算逻辑较简单，主要受访存带宽影响，属于Memory Bound类型，因此每个线程块的线程数可配置为2048
- 该场景下由于每个线程处理1个元素，线程块数量计算逻辑为：元素总数为2097152、每个线程块设置包含2048线程，因此线程块数量为`2097152 / 2048 = 1024`
- 通过`SCENARIO_NUM`取值为6控制`<<<...>>>`设置线程块数量为1024，执行单线程单数据场景

**关键代码**：

```cpp
// 基础写法：每个线程只处理1个元素，无for循环
uint32_t global_idx = blockIdx.x * blockDim.x + threadIdx.x;

uint32_t gather_idx = index[global_idx];
if (gather_idx >= input_total_length) { gather_idx = 0; }
output[global_idx] = input[gather_idx];
```

**性能数据**：

| Task Duration（us） |
| :---------------: |
|      78.289       |

#### 单线程多数据

**优化目标**：在语义不变的前提下，**每个线程处理多个元素（单线程多数据）**，通过减少线程块数量，来减少调度开销。

**核心实现**：

- 通过`SCENARIO_NUM`取值为7或8控制`<<<...>>>`设置两种线程块数量配置：64 / 32，执行单线程多数据场景，核函数`gather_strided_kernel`每个线程从全局编号`global_idx`作为起始，以总线程数为步长循环处理多个元素
- 每个线程块的线程数设置为2048；线程块数量动态配置：
  - 线程块数量为64：总线程数为131072，即`64 * 2048`，每个线程处理16个元素
  - 线程块数量为32：总线程数为65536，即`32 * 2048`，每个线程处理32个元素

**关键代码**：

```cpp
uint32_t global_idx = blockIdx.x * blockDim.x + threadIdx.x;
uint32_t stride = gridDim.x * blockDim.x;

for (uint32_t i = global_idx; i < index_total_length; i += stride) {
    uint32_t gather_idx = index[i];
    if (gather_idx >= input_total_length) { gather_idx = 0; }
    output[i] = input[gather_idx];
}
```

**性能数据**：

| 线程块数量 | Task Duration（us） |
| :-----: | :---------------: |
|   64    |      58.583       |
|   32    |      110.540      |

#### Case 1 三种配置性能对比

| 写法 | 线程块数量 | Task Duration（us） |
| :--: | :-----: | :---------------: |
| 单线程多数据 | 32 | 110.540 |
| **单线程多数据** | **64** | **58.583** | 
| 单线程单数据 | 1024 | 78.289 |

下图直观展示了上表中Task Duration随线程块数量的变化趋势：

<img src="figures/fig1_case1_large_curve.png" alt="Case 1 大数据量场景（2097152） Task Duration 随线程块数量的变化" width="600" />

**分析**：
- 配置64个线程块时，相比1024个线程块的单线程单数据写法，Task Duration从78.289us降低到58.583us，耗时下降约25.2%，性能提升约1.34倍。Gather算子读写总量和线程块的线程数均不变，提升主要来自线程块数量减少后，启动和调度的线程块数量显著减少所带来的收益。
- 配置64个线程块时，相比32个线程块，Task Duration从110.540us降低到58.583us，耗时下降约47.0%，性能提升约1.89倍。同样的单线程多数据写法下，64个线程块具备更高的并行度，每个线程块承担的工作量更少；虽然线程块数量增加会带来更多启动和调度开销，但这部分开销小于并行度提升带来的收益，因此整体耗时更低。


**结论**：
- 受硬件资源限制，实际可并行执行的物理核数量存在上限。本样例实测环境的物理核数为64，可参考`grid_config.asc`中的`query_aiv_core_num`接口查询。由于一个物理核同一时刻只能驻留并执行一个线程块，当线程块数量为1024时，超出物理核数量的线程块需要等待前序线程块执行完成后再调度，额外的启动和调度固定开销会明显增加；当线程块数量为32时，只使用了一半物理核，空闲物理核较多，并行度不足。
- 因此，大数据量场景下应优先让线程块数量匹配实际物理核数。本样例中线程块数量为64既避免了线程块数量为1024带来的大量线程块调度开销，又避免了线程块数量为32时并行度不足、单线程块工作量过重，因此是Case 1中的最优配置。

### Case 2：小数据量场景，Gather算子配置不同线程块数量性能对比

**样例目标**：选取一个小数据量场景`8*2048`，即16384个元素，对比该场景下执行Gather算子，配置不同线程块数量的性能差异。

**核心实现**：

- 通过`SCENARIO_NUM`取值为9/10/11/12/13控制`<<<...>>>`设置五种线程块数量配置：4 / 8 / 16 / 32 / 64，核函数与Case 1的单线程多数据写法完全相同，仅将数据量调整为`8*2048`，即16384个元素
- 每核处理的元素数量 = 16384 / 线程块数量：4核4096、8核2048、16核1024、32核512、64核256
- 每线程块线程数按每核实际元素数自适应：4 / 8核每核元素数≥2048，线程数取上限2048；16 / 32 / 64核每核元素数小于2048，线程数自适应降到每核实际元素数（1024 / 512 / 256），消除空转线程，具体配置如下表：

  | 线程块的数量 | 每个线程块的线程数 | SCENARIO_NUM |
  | :----------: | :----------------: | :----------: |
  |      4       |        2048        |      9       |
  |      8       |        2048        |      10      |
  |      16      |        1024        |      11      |
  |      32      |        512         |      12      |
  |      64      |        256         |      13      |

**性能数据**：

| 线程块数量 | Task Duration（us） |
| :--------: | :---------------: |
|     4      |      10.831       |
|     8      |       6.569       |
|     16     |       4.500       |
|   **32**   |     **3.696**     |
|     64     |       4.133       |

下图直观展示了上表中Task Duration随线程块数量的变化趋势：

<img src="figures/fig2_case3_16384_curve.png" alt="Case 2 小数据量场景（16384） Task Duration 随线程块数量的变化" width="600" />


**分析**：

- 折线呈“先降后升”的趋势，**本样例实测最优线程块数量为32**，而非使用全部物理核的64。这里的32只是该数据量下的实测结果，并不是小数据量场景的通用值。
- 线程块数量从4依次增加到8、16、32时，Task Duration持续下降（10.831 → 6.569 → 4.500 → 3.696）。这是因为线程块中的线程并不是全部在并行执行，而是以warp为单位调度执行，单核上warp过多时会产生排队等待，线程块数量较小时，每个线程块需要启动更多线程，等待时间更明显；增加线程块数量可以降低单个线程块的线程数并提高并行核数，因此收益明显。
- 线程块数量从32进一步翻倍到64时，Task Duration从3.696us升到4.133us，**耗时上升约11.8%**。此时每核工作量已很小、有效计算时间短，新增线程块带来的调度开销反而超过并行收益。
- 与Case 1的大数据量场景对照：同样的kernel，数据量缩小后最优线程块数量随之改变，说明“最优线程块数量”没有可照搬的值，必须实测。

**结论**：小数据量场景无法直接沿用“使用全部物理核”的经验，必须参考本Case的方式设置多种线程块数量实测。

## 性能对比总结

各Case的性能数据见前述对应表格，本节按大/小数据量场景汇总跨Case的横向规律：

本样例以数据量相对硬件并行能力的比例来区分大数据量和小数据量。参考量级为“物理核数 × 单核最大线程数”，当前实测环境为64个物理核、每核最大2048个线程，即参考量级为131072个元素。Case 1的数据量为2097152，远大于该参考量级，因此作为大数据量场景；Case 2的数据量为16384，小于该参考量级，因此作为小数据量场景。该划分仅用于说明本样例中的调优现象，实际业务应结合目标硬件、算子特征和实测结果判断。

### 大数据量场景

- **结论**：大数据量场景下应优先让线程块数量匹配物理核数。本样例中`1024*2048`对应2097152个元素，线程块数量为64时Task Duration为58.583us，低于线程块数量为1024时的78.289us和线程块数量为32时的110.540us，因此线程块数量为64是该Case中的最优配置。
- **原因**：线程块数量远超物理核数时（如1024），需要启动和调度的线程块数量较多，启动和调度固定开销明显累积（Case 0：线程块数量为1024时调度基线为16.422us，约为线程块数量为64时的2.173us的7.6倍）；线程块数量过小时（如32），只用到一半物理核，并行核数不足，单线程块工作量翻倍。线程块数量为64时，在物理核利用率与调度开销之间取得较好平衡。

### 小数据量场景

- **结论**：小数据量场景的最优线程块数量随数据量变化，不能直接沿用“使用全部物理核”的经验，必须设置多种线程块数量实测。本样例中`8*2048`对应16384个元素，实测最优线程块数量为32，并非物理核数64，且该值仅对本数据量成立。
- **原因**：数据量较小时，每核分到的工作量很小、有效计算时间短，此时继续增加线程块数量（如从32到64，耗时上升约11.8%），新增线程块带来的启动和调度固定开销会超过并行收益。

## 调优建议

1. **大数据量场景，优先让线程块数量匹配物理核数**：线程块数量过小会导致并行核数不足、单线程块工作量变重；线程块数量过大则会引入更多线程块启动和调度开销。
2. **小数据量场景需实测线程块数量**：当数据量较小、每核工作量很小时，启动更多线程块带来的调度固定开销会超过收益，最优线程块数量不一定是物理核数。应通过msprof工具实测性能，选择Task Duration较低的线程块数量配置。

## 编译运行

在本样例根目录下执行如下步骤，编译并执行样例。

- 配置环境变量
   请根据当前环境上CANN开发套件包的[安装方式](../../../../../docs/quick_start.md#prepare&install)，配置环境变量。
  ```bash
  source ${install_path}/cann/set_env.sh
  ```

  > **说明：** `${install_path}` 为CANN包安装目录，未指定安装目录时默认安装至 `/usr/local/Ascend` 下。

- 样例执行
  在本样例目录下执行如下命令。
  ```bash
  SCENARIO_NUM=7                                                                     # 选择执行场景，可选1-13
  mkdir build && cd build;                                                           # 创建并进入build目录
  cmake -DCMAKE_ASC_ARCHITECTURES=dav-3510 -DSCENARIO_NUM=$SCENARIO_NUM ..;make -j;  # 编译工程
  ./grid_config                                                                      # 执行样例
  ```

- 编译选项说明

  | 选项                       | 可选值      | 说明                                                          |
  | -------------------------- | ----------- | ------------------------------------------------------------- |
  | `CMAKE_ASC_ARCHITECTURES`  | `dav-3510`  | NPU架构：本样例仅支持dav-3510（Ascend 950PR/Ascend 950DT）    |
  | `SCENARIO_NUM`             | `1`-`13`    | 样例类型，默认为1                                             |

  执行结果如下，说明精度对比成功。

  ```text
  [Success] Case accuracy is verification passed.
  ```

## 性能分析

使用`msprof`工具获取详细性能数据：

```bash
msprof op ./grid_config   # 分析case的性能
```

命令完成后，会在默认目录下生成以“OPPROF_{timestamp}_XXX”命名的文件夹，性能数据文件夹结构示例如下：

```text
├──dump                       # 原始的性能数据，用户无需关注
├──ArithmeticUtilization.csv  # cube/vector指令cycle占比
├──L2Cache.csv                # L2 Cache命中率
├──Memory.csv                 # UB，L1和主存储器读写带宽速率
├──MemoryL0.csv               # L0A，L0B，和L0C读写带宽速率
├──MemoryUB.csv               # Vector和Scalar到UB的读写带宽速率
├──OpBasicInfo.csv            # 算子基础信息
├──PipeUtilization.csv        # 采集计算单元和搬运单元耗时和占比
├──ResourceConflictRatio.csv  # UB上的bank group、bank conflict和资源冲突率在所有指令中的占比
└──visualize_data.bin         # MindStudio Insight呈现文件
```
